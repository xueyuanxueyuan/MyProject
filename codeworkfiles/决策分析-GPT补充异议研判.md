## 2. 核心异议的专业研判

### 2.1 补充意见 1：“业务禁重试” + “技术自动重投”
- **原委**：我们之前为了砍掉复杂的业务状态机，直接规定“失败就办结，重发新单”。
- **GPT的毒辣之处**：它指出了“业务失败（比如银行拒绝）”和“技术抖动（比如网络超时 504）”的区别。如果是网络闪断，直接把单子置为失败并让业务员重发，会导致业务员天天骂系统垃圾。
- **我的研判**：**必须采纳**。前端和业务状态依然保持没有“重试”按钮（业务层干净）；但在后端代码里，必须对“超时类”错误利用消息队列或 XXL-JOB 进行同幂等键的 N 次自动重投。N 次之后还失败，再正式置为“失败”并释放缺口。这才是成熟大厂的解法。

### 2.2 补充意见 2：状态模型双字段拆分
- **原委**：我们之前为了前端展示简单，把部分成功合并到了“调拨完成”里。
- **GPT的毒辣之处**：它站在了“DBA 和审计对账”的视角。如果只有一个字段，月底财务跑报表算“到底有多少钱调拨失败了”，一条 SQL 根本写不出来。
- **我的研判**：**必须采纳**。前端展示可以只显示一个文案，但数据库里必须是两个字段：`order_close_status`（是否办结：是/否）和 `order_result_status`（结果：全成功/部分成功/全失败）。这样无论是出报表还是对账，逻辑清晰无歧义。

### 2.3 补充意见 3：金额校验的三层防御
- **原委**：我们之前加了 `单行金额 <= 缺口资金` 的拦截。
- **GPT的毒辣之处**：黑客或内鬼思维。如果总缺口是 100 万，我建 3 条明细，每条都是 100 万（单行校验通过），这单据总额就变成了 300 万，直接把账户掏空。
- **我的研判**：**必须采纳**。在提交时，除了单行校验，必须增加“同付款账号汇总求和 <= 实时可用余额”以及“整单汇总 <= 总缺口”的汇总级拦截。这在资损防控上叫做“兜底防线”。

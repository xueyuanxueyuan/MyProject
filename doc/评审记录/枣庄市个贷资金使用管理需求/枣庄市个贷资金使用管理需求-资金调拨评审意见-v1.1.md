# 枣庄市个贷资金使用管理需求-资金调拨评审意见-v1.1

## 1. 评审目标

基于 v1.0 评审文档进行二次批判性评审，输出：

- 对 v1.0 的认可点与理由
- 对 v1.0 的不认可点与修正建议
- 对需求分析文档的落地增补结果

## 2. 总体结论

- 结论：v1.0 的结构化质量较高，可作为评审会输入基线。
- 采纳策略：采纳其“分层治理 + 四清单化验收”主框架。
- 修正策略：对个别表述不一致项进行纠偏，避免状态/操作口径歧义进入设计与测试阶段。

## 3. 认可项（采纳）

### 3.1 结构与方法

- 认可 A/B/C/D 四清单拆分方式：
  - A：硬需求可追溯
  - B：状态流转
  - C：字段与校验
  - D：实现增强建议
- 认可“硬需求与增强建议分层”原则，有助于避免范围蔓延与验收争议。

### 3.2 业务口径

- 认可资金不足分流链路：先放款可满足部分，再对不足部分发起资金调拨。
- 认可集中调拨口径：当前余额不足账户集中申请。
- 认可按钮可点击条件与可用放款金额显示口径：
  - 可用放款金额 = 当前账户余额 - 账户备付金
  - 小于 0 显示 0
  - 可用放款金额不等于 0 才允许放款确认
- 认可调拨支付结果字段化建议：收/付款行与账号、金额、状态应固定展示。
- 认可附件追溯要求：未放款清册附件应可在详情或审批关联视图中查询。

## 4. 不认可项（修正）

### 4.1 状态-操作口径存在局部不一致

- 问题 1：v1.0 的状态流转表中“申请”状态未列“修改”，与模块操作定义不一致。
- 修正：申请状态应允许“修改、提交审批、取消申请、详情”。

- 问题 2：v1.0 的状态流转表中“驳回”状态未列“取消申请”，与原文“驳回后可取消申请”不一致。
- 修正：驳回状态应允许“修改、取消申请、详情”。

### 4.2 个别文案命名需统一

- 问题：文档中“发放确认/放款确认”混用，可能导致接口命名与测试用例命名分裂。
- 修正：统一采用“放款确认”作为业务动作名称。

### 4.3 D 清单边界需继续强调

- 问题：虽已分层，但评审执行时仍可能把幂等、强校验等工程项当作硬需求。
- 修正：在评审纪要中明确“D 清单默认非阻断验收项，除非产品/架构会签升级为当期范围”。

## 5. 已回写到需求分析文档的增补项

- 新增“评审采纳补充（基于 v1.0 评审意见）”章节。
- 新增“操作权限矩阵（状态 x 操作）”并修正申请/驳回状态的可操作项。
- 新增“评审分歧保留项”，明确哪些属于增强建议而非硬约束。

## 6. 评审后统一口径（建议作为会议结论）

- 口径 1：A 清单是硬验收基线，D 清单是工程增强项候选池。
- 口径 2：状态操作以“状态 x 操作矩阵”为准，避免前后端理解偏差。
- 口径 3：调拨支付先落实原文字段，扩展字段按版本范围评估。
- 口径 4：附件追溯入口需在详细设计明确到页面位置。

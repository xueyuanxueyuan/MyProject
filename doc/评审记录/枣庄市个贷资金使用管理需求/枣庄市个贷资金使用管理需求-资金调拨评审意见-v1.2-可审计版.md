# 枣庄市个贷资金使用管理需求-资金调拨评审意见-v1.2-可审计版

## 1. 本版目的

本文件用于对上一版（`v1.1`）的结论进行“可审计化”处理：  
1) 所有涉及原需求口径的结论，均给出对应的原文章节与关键句证据；  
2) 所有无法由原文直接推出的内容，必须标注为“待业务确认/推导/建议”，禁止当作硬需求直接验收；  
3) 本文件不修改原始需求文档（仅给出评审口径与验收建议），避免“已回写”引发审计争议。

## 2. 证据来源范围（原文章节）

原始需求文档：`doc/需求文档/枣庄市个贷资金使用管理需求.md`

## 3. 硬需求口径审计（与原文对照）

### 3.1 资金确认阶段（`1.1.1.2`）硬需求

1) `可用放款金额`口径与展示约束  
- 证据（原文）：`可用放款金额 = 当前账户余额 - 账户备付金，若<0 则显示0`（`1.1.1.2.4`）

2) `放款确认`按钮可点击条件  
- 证据（原文）：`可点击放款确认（可用放款金额<>0的可点击放款确认）`（`1.1.1.2.4`）

3) 不足时链路与附件  
- 证据（原文）：  
  - `对于余额不足未能放款的部分，可发起资金调拨申请流程，并将未放款的放款申请清册作为资金调拨附件`（`1.1.1.2.1 / 1.1.1.2.4`）  
  - `将当前全部余额不足的账户集中进行资金调拨申请`（`1.1.1.2.1 / 1.1.1.2.4`）

4) 资金调拨申请-调拨清册字段（反显/可编辑/默认值）
- 证据（原文，`1.1.1.2.5`）：  
  - 调拨收款银行/账号：`反显，不可修改`  
  - 调拨付款银行：`根据付款账号选择反显，不可修改`  
  - 调拨付款账号：`下拉列表`，默认：`资金余额表中可用余额最大银行`，`可修改`  
  - 调拨金额：`文本框`，默认：`缺口资金`，`可修改`  
  - 操作：`增加行、删除行`

### 3.2 贷款资金调拨管理阶段（`1.1.1.3`）硬需求

1) 列表操作集合  
- 证据（原文，`1.1.1.3.4`）：  
  - `操作（修改、提交审批、详情、调拨支付、取消申请）`

2) 修改/再提交/取消的触发条件  
- 证据（原文，`1.1.1.3.4`）：  
  - `若资金调拨审批被驳回后，可在此功能中进行修改，修改后可重新提交审批，也可以取消申请。`
  - `1)修改...` `2)提交审批...` `3)取消申请：驳回后，可取消申请，取消后不可修改、提交审批，申请作废。`

3) 调拨支付触发条件  
- 证据（原文，`1.1.1.3.4`）：  
  - `若资金调拨审批通过后，可在此功能中实现资金调拨支付。`

4) 8 个资金调拨状态语义（用于状态机）
- 证据（原文，`1.1.1.3.4`）：  
  1) 申请：驳回后并已完成修改未重新提交前  
  2) 审批中：提交审批后、完成终极审批前  
  3) 驳回：审批驳回后，未修改前  
  4) 作废：审批驳回后，取消申请  
  5) 待调拨：完成终极审批，未发起调拨支付申请  
  6) 调拨确认：发起调拨支付申请后、未完成调拨支付确认前  
  7) 结算中：调拨支付确认后，在返回结算状态前  
  8) 调拨完成：调拨结算全部交易成功

5) 调拨支付展示字段（用于验收测试）
- 证据（原文，`1.1.1.3.5`）：  
  - 收款银行、收款账号、付款银行、付款账号、调拨金额、状态

## 4. 状态 x 操作矩阵（证据绑定 + 推导标注）

说明：原文给出了“操作集合”和“触发条件”，未直接给出“每个状态对应允许哪些按钮”的逐格表。  
因此本矩阵采用如下规则生成：  
- 能由原文的触发条件直接推出的，标为“直接证据”；  
- 需要将触发条件与状态语义组合的，标为“推导”；  
- 若无法推出，标为“待确认”。

| 状态 | 允许操作（建议） | 证据/推导来源 |
| --- | --- | --- |
| 申请 | 修改、提交审批、取消申请、详情 | 直接推导：`申请=驳回后且已完成修改未重新提交` + 原文`驳回后可修改、修改后可重新提交审批，也可以取消申请`（`1.1.1.3.4`） |
| 审批中 | 详情 | 推导：原文描述仅在“驳回后可修改/取消、通过后可调拨支付”，审批中未列修改/取消入口（`1.1.1.3.4`） |
| 驳回 | 修改、取消申请、详情 | 推导：状态语义`驳回=审批驳回后未修改` + 原文`驳回后可修改...也可以取消申请`（`1.1.1.3.4`） |
| 作废 | 详情 | 推导：`作废=审批驳回后取消申请`且原文`取消后不可修改、提交审批`（`1.1.1.3.4`） |
| 待调拨 | 调拨支付、详情 | 推导：状态语义`待调拨=完成终极审批未发起调拨支付` + 原文`审批通过后可实现调拨支付`（`1.1.1.3.4`） |
| 调拨确认 | 详情 | 推导：原文未列在“确认前”仍允许再次发起支付的入口，仅描述展示调拨结果/失败可修复（`1.1.1.3.4`） |
| 结算中 | 详情 | 推导：原文描述偏“返回结算状态前”，未列额外可操作项（`1.1.1.3.4`） |
| 调拨完成 | 详情 | 推导：终态（`调拨完成：全部交易成功`）（`1.1.1.3.4`） |

## 6. D 清单升级机制（明确“是否阻断”）

为了避免将工程项误当作硬需求，本版明确以下决策机制：

| D 清单项 | 默认验收属性 | 何时升级为阻断 | 决策主体 |
| --- | --- | --- | --- |
| 幂等/防重（支付逐条幂等、重试一致性） | 非阻断验收（建议项） | 出现“重复扣划/重试不一致”类重大风险；或架构会签纳入当期范围 | 产品/架构 |
| 失败治理（失败原因、可修复重试） | 非阻断验收（建议项） | 需要对外审计合规（或验收用例强制要求） | 产品/测试负责人 |
| 并发与一致性（支付前二次校验） | 非阻断验收（建议项） | 会导致明确的超额/越权风险且需强制拦截 | 架构/安全负责人 |
| 审计与追溯（全链路留痕、附件可追溯） | 非阻断验收（建议项） | 监管/内控要求强制落地（在会签纪要中写明） | 内控/合规 |

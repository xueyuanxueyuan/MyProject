# 手续费预提调整优化方案第二轮对抗性评审

## 1. 评审对象
- 回应文档：`doc/操作记录/手续费预提调整优化方案-对抗性评审回应.md`
- 评审目标：判断回应方案是否已从“原则性错误”修正为“可安全落地”，并评估距离 `95+` 还差哪些关键补强
- 评审结论：**已显著改善，但仍不建议按当前骨架直接实施**
- 综合评分：**86 / 100**

## 2. 本轮认可项
- 已明确撤回“直接信任前端 DTO 组凭证”的错误结论，恢复 `mxMap` 作为数据库权威快照来源。
- 已明确放弃不带并发保护的 `updateBatchById` 伪优化，方向回到“正确性优先”。
- 已补充重复 `mxId` 校验和门禁异常阻断思路，问题识别明显更完整。
- 已接受“本业务量级下，一致性优先于极限性能”的原则，这一点非常关键。

## 3. 主要问题项

### 3.1 最严重问题：开始更新后仍然 `return R.fail(...)`，事务未必回滚
- 方案骨架仍然在更新循环中使用：
  - 金额不一致时 `return R.fail(...)`
  - 条件更新失败时 `return R.fail(...)`
- 但原方法是 `@Transactional(rollbackFor = Exception.class)` 模式，**Spring 默认只对异常回滚，不会因为返回 `R.fail` 自动回滚**。
- 这意味着如果前几条已更新成功，后面某一条触发 `return R.fail`，前面已成功的更新很可能会被提交，形成**部分成功、整体失败**的脏状态。
- 这是当前回应稿中最大的剩余风险，严重程度高于一般的边界处理问题。
- 处理建议：
  - 一旦进入“开始写库”的阶段，后续任何失败都必须 `throw BizException(...)`；
  - 或者更稳妥地先完成全部校验与待更新命令构建，再统一落库。

### 3.2 条件更新对 `null` 金额处理不严谨，可能误伤历史数据
- 回应稿中使用：
  - `BigDecimal currentSxf = currentMx.getSxf() == null ? BigDecimal.ZERO : currentMx.getSxf();`
  - `updateWrapper.eq(SxfytMxDO::getSxf, currentSxf)`
- 如果数据库里的原值真的是 `null`，这里会把条件写成 `sxf = 0`，而不是 `sxf is null`。
- 结果是：
  - Java 比对阶段把 `null` 当 `0` 看，校验通过；
  - SQL 更新阶段却因为 `where sxf = 0` 匹配不到原始 `null` 行，导致误判为“被他人修改”。
- 处理建议：
  - 比对可以继续按“空视零”；
  - 但更新条件必须基于数据库原始值：
    - 原始值为 `null` 时用 `isNull`；
    - 原始值非 `null` 时再用 `eq(rawValue)`。

### 3.3 仍未显式覆盖 `mxId` 为空 / `mxIds` 无效的场景
- 回应稿补了重复校验，但没有显式补：
  - 单个 `item.mxId == null`
  - `itemList` 非空但提取出的 `mxIds` 无效
- 当前写法：
  - `map(...getMxId).distinct().count()`
  - `map(...getMxId).toList()`
- 对“单个 `null` 且无重复”的场景，防重校验不会拦住，后续 `listByIds(mxIds)` 仍可能接收到非法参数。
- 处理建议：
  - 在最前面逐条校验 `mxId` 非空；
  - 明确区分“请求缺字段”和“数据库无此记录”两类错误。

### 3.4 金额平衡校验放在更新之后，不是最优落点
- 回应稿把金额汇总和更新放在同一循环里，但在循环结束后才做总额平衡校验。
- 虽然其后若 `throw BizException` 理论上可回滚，但这会带来：
  - 不必要的数据库写入与锁竞争；
  - 让代码正确性依赖事务回滚，而不是前置校验。
- 处理建议：
  - 同一轮遍历里可以同时完成金额累计、快照校验、差额收集；
  - 但应先把“待更新命令”缓存起来；
  - 在 `totalYtqje == totalYthje` 验证通过后，再执行真正的 update。

### 3.5 “强门禁异常即阻断”方向正确，但仍需业务口径确认
- 从技术风险控制角度，我认可“查门禁失败则不允许继续保存”。
- 但这一点属于业务行为变化：
  - 旧逻辑是依赖失败时放行；
  - 新逻辑是依赖失败时阻断。
- 如果要进入高分落地方案，应该在方案文档里补一句：
  - “已与业务确认该校验属于强门禁，依赖失败时应阻断保存”。
- 没有业务口径确认时，这一点仍属于“技术上合理、业务上待确认”。

### 3.6 伪代码异常写法与现有项目风格不一致
- 回应稿中写了：`throw new BizException("调整后预提金额合计必须等于调整前金额合计");`
- 但当前类里的现有写法更接近：
  - `throw new BizException(ResultErrorCodeEnum.FAIL.getCode(), "...")`
- 如果这是最终实施依据，建议保持和现有项目异常构造风格一致，避免实现时再返工。

### 3.7 记账通知数据过滤与正负方向规则仍需显式固化
- 当前线上实现里，记账前已经有一层过滤：
  - 仅当 `ythje - currentSxf != 0` 时，才把明细加入 `tzItemList`；
  - 记账明细中的 `zje` 直接取 `tzce`，即“调增为正、调减为负”。
- 这说明现有业务口径本身是清楚的，但回应方案里还没有把这条规则提升为“必须检查的显式约束”，风险在于后续重构时可能被无意改坏。
- 必须补充两类检查：
  - **未作调整的明细不得通知财务记账**：`tzce == 0` 的明细不得进入 `tzItemList`，也不得生成 `CwjzYwlsMxDTO`；
  - **调整差额正负号必须正确**：`tzce > 0` 表示调增，传正数；`tzce < 0` 表示调减，传负数。
- 特别要防止两种错误实现：
  - 为了计算主表 `ywfse` 使用了 `abs()`，结果顺手把明细 `zje` 也写成绝对值，导致调减也传正数；
  - 先过滤出有差额项，后续在生成凭证时又重新组装，结果把 `tzce == 0` 的项重新混入记账列表。
- 处理建议：
  - 在“差额计算与待记账项收集”阶段就明确：
    - `tzce == 0`：只允许更新预提数据，不允许进入记账通知；
    - `tzce != 0`：才允许进入 `tzItemList`；
  - 在记账组装私有方法中再做一次防御性过滤，避免调用方误传；
  - 在方案中明确区分：
    - 明细 `zje`：保留有符号差额；
    - 主表 `ywfse`：使用差额绝对值汇总后的单边金额。

## 5. 距离 95+ 还差什么
- 第一，**把所有“开始写库后的失败返回”改成抛异常**。
- 第二，**把 `sxf` 条件更新改成基于数据库原始值判断 `isNull/eq`**，不要直接用空转零后的值做 `where`。
- 第三，**显式补 `mxId` 非空校验和无效 `mxIds` 校验**，不要只做重复检测。
- 第四，**先完成总额平衡与待更新数据准备，再执行数据库更新**，避免先写后判。
- 第五，**在方案中写明强门禁口径已经过业务确认**。
- 第六，**把“零差额不通知记账、调减传负数/调增传正数”写成显式检查规则**，并在记账组装方法里做二次防御。

## 6. 建议后的更稳妥骨架
- 阶段一：前置校验
  - `itemList` 非空
  - 每条 `mxId` 非空
  - `mxId` 不重复
  - 门禁校验通过
  - 影像校验通过
- 阶段二：查库构建 `mxMap`
  - 校验明细全部存在
- 阶段三：一次遍历只做计算与命令准备
  - 校验数据库当前金额与前端调整前金额一致
  - 累计总额
  - 计算差额
  - 仅当 `tzce != 0` 时收集待记账项
  - 组装待更新命令，但**暂不执行 update**
- 阶段四：总额平衡校验
  - 不通过则直接抛异常，尚未写库
- 阶段五：执行条件更新
  - `where id = ? and (sxf is null / sxf = 原值)`
  - 任一失败则抛异常回滚
- 阶段六：生成记账凭证
  - 关键字段全部从 `mxMap` 取
  - 明细 `zje` 必须保留 `tzce` 原始符号
  - 主表 `ywfse` 才使用绝对值汇总口径

## 7. 最终意见
- 这份回应稿相比上一版有明显进步，已经从“方向跑偏”回到了“基本走在正路上”。
- 但当前版本还不能打到 `95+`，核心原因不是思路不对，而是**事务回滚语义和 `null` 原值条件更新**两个点仍然带着真实事故风险。
- 只要把这两个关键点补齐，再把 `mxId` 非空和总额校验前置补完整，我认为这份方案就能进入高分可实施区间。

## 8. 最终修订要求清单（请按此清单重写方案）

### 8.1 必须保留的正确设计
- 保留 `mxMap`，并明确其职责是“数据库权威快照字典”，不是冗余结构。
- 记账关键维度字段必须以数据库明细为准，尤其是：
  - `zhbh`
  - `yhdm`
  - `jbjgbh`
  - `jbjgmc`
- 前端 DTO 只允许承担“变更输入”的职责，不得作为账务凭证维度来源。

### 8.2 必须补齐的前置校验
- `itemList` 非空。
- 每条 `mxId` 非空。
- `mxId` 不重复。
- 查询得到的 `mxIds` 非法时要明确拦截，不能把风险留给底层框架处理。
- `mxId` 对应明细必须全部存在，不存在时要和“请求缺字段”区分报错。
- 排名更新状态查询属于强门禁时，依赖异常必须阻断保存，不允许吞异常后放行。
- 影像材料校验必须在写库前完成。

### 8.3 必须调整的事务与失败处理方式
- 一旦进入写库阶段，任何失败都必须抛异常，不能再 `return R.fail(...)`。
- 方案中所有“保存失败”“金额已变化”“并发冲突”“总额不平”等分支，在写库阶段都应使用项目现有风格的 `BizException`。
- 不要把数据正确性建立在“前面已经写了，后面失败再靠事务回滚兜底”的脆弱路径上；更稳妥的方式是先完成校验和命令准备，再执行更新。

### 8.4 必须修正的并发更新口径
- Java 层金额比较可以“空视零”。
- SQL `where` 条件不能直接复用空转零后的值：
  - 数据库原始值为 `null` 时，更新条件必须使用 `isNull`；
  - 数据库原始值非 `null` 时，更新条件才使用 `eq(rawSxf)`。
- 如果条件更新失败，应认定为并发修改或数据变化，直接抛异常回滚。
- 在没有明确乐观锁版本字段支持前，不建议再次引入 `updateBatchById` 替代当前条件更新语义。

### 8.5 必须前置的金额处理顺序
- 在一次遍历中完成：
  - 当前金额一致性校验；
  - 总额累计；
  - 差额计算；
  - 待记账项收集；
  - 待更新命令组装。
- 但此阶段**只做准备，不真正写库**。
- 必须先通过“调整前合计 = 调整后合计”的平衡校验，再执行数据库更新。

### 8.6 必须固化的记账通知规则
- **零差额不记账**：`tzce == 0` 的明细不得进入 `tzItemList`，也不得生成 `CwjzYwlsMxDTO`。
- **调增传正数**：`tzce > 0` 时，记账明细金额必须为正。
- **调减传负数**：`tzce < 0` 时，记账明细金额必须为负。
- **明细与主表口径分离**：
  - 明细 `zje` 必须保留差额原始符号；
  - 主表 `ywfse` 才允许按差额绝对值汇总计算单边金额。
- 在主流程收集 `tzItemList` 时做一次过滤，在记账组装私有方法中再做一次防御性过滤，避免重构时把零差额项重新混入记账列表。

### 8.7 必须保持的项目风格一致性
- 异常构造风格应与当前类保持一致，优先使用现有业务码风格。
- 文案要区分以下场景，避免全部混成“请刷新后重试”：
  - 请求参数缺失；
  - 明细不存在；
  - 金额已被他人修改；
  - 门禁校验失败；
  - 影像校验失败；
  - 财务记账失败。
- 若方案要声明“强门禁异常即阻断”，应补一句“已与业务确认该规则成立”。

## 9. 高分验收标准（95+）
- 同时满足以下条件，我认为方案可以进入 `95+`：
- 不再错误削弱 `mxMap` 或数据库快照口径。
- 不再信任前端 DTO 作为账务维度数据源。
- 不再使用“写库后 `return R.fail`”这种会留下部分成功风险的模式。
- 条件更新正确处理数据库原始 `null` 值语义。
- 显式补齐 `mxId` 非空、重复、失效三类校验。
- 总额平衡校验前置到真正写库之前。
- 记账通知规则明确写出“零差额不记账、调增正数、调减负数”。
- 方案文字中明确主表 `ywfse` 与明细 `zje` 的口径差异。
- 强门禁策略有明确业务确认说明。

## 10. 禁止踩坑项
- 不要把 `mxMap` 删除后再回头信任前端字段。
- 不要把 `updateBatchById` 包装成“天然安全的性能优化”。
- 不要把 `abs()` 用到记账明细 `zje` 上。
- 不要在主流程中过滤了零差额项，却在记账私有方法里重新把它们带回来。
- 不要在已执行部分 update 后再用 `return R.fail` 结束事务。
- 不要把数据库 `null` 金额直接等价成 SQL 条件里的 `0`。

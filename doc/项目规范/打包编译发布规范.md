# Gjj 项目打包编译发布规范

## 1. 概述
本规范旨在标准化 Gjj 仓库中各地区项目（如 `zaozhuang`、`linyi` 等）的前后端代码拉取、环境配置、编译打包及制品发布的完整流程，确保操作的一致性与可复现性。

## 1.1 指令语义强约束
- 当需求中仅包含“编译打包”时，只执行编译打包与本地制品归档，不执行远程上传与服务重启。
- 当需求中明确包含“发布”时，必须调用远程部署流程执行上传与部署。
- 发布支持两种模式：
  - 指定单一程序发布：仅上传用户指定包，并仅重启该包对应服务。
  - 发布目录全量发布：上传 `<地区根目录>/release` 下全部包，并按部署清单批量重启服务。

## 2. 目录结构约定
以地区名称作为项目根目录（例如 `zaozhuang`）：
- **后端代码**：`<地区根目录>/IdeaProjects/capinfo-gjj-busi-jshs`
- **前端代码**：`<地区根目录>/WebstormProjects/capinfo-gjj-frontend-jshs-gm`
- **发布制品**：`<地区根目录>/release`

## 3. 工具链依赖
统一使用 `vfox` 管理多版本 SDK：
- **后端**：`java@17.0.2+8` 和 `maven@3.9.14`
- **前端**：`nodejs@16.20.2`（Vue2及老旧 node-sass 依赖要求）

## 4. 后端打包与发布流程

### 4.1. 代码更新
```bash
cd <地区根目录>/IdeaProjects/capinfo-gjj-busi-jshs
git pull
```

### 4.2. 环境配置与验证
```bash
export JAVA_HOME="/home/source/.vfox/cache/java/v-17.0.2+8/java-17.0.2+8"
export PATH="${JAVA_HOME}/bin:/home/source/.vfox/cache/maven/v-3.9.14/maven-3.9.14/bin:$PATH"

java -version  # 应输出 17.0.2
mvn -v         # 应输出 3.9.14
```

### 4.3. 编译打包
执行 Maven 打包命令，跳过测试。若遇到 Aliyun 镜像被拦截问题，确保指定或修改包含正确 `<mirror>` 的 `settings-tsp.xml`。
```bash
mvn clean package -DskipTests -s settings-tsp.xml
```

### 4.4. 归档制品
1. 在 `<地区根目录>` 下创建 `release` 文件夹。
2. 归档前先清空 `release` 目录，确保本次发布包全部为最新构建产物。
3. 将所有生成的 `-app.jar` 迁移至 `release` 目录。
```bash
mkdir -p <地区根目录>/release
find <地区根目录>/release -mindepth 1 -delete
find <地区根目录>/IdeaProjects/capinfo-gjj-busi-jshs -name "*-app.jar" -type f -exec cp {} <地区根目录>/release/ \;
```

### 4.5. 发布动作（仅在用户明确要求“发布”时执行）
1. 指定单一程序发布（示例）：
```bash
python3 /home/source/Jetbrains/Probject/Gjj/scripts/remote_deploy.py <region> <file_name>
python3 /home/source/Jetbrains/Probject/Gjj/scripts/remote_exec.py <region> "kubectl rollout restart deployment <deployment_name> -n <namespace> && kubectl rollout status deployment <deployment_name> -n <namespace> --timeout=180s"
```
2. 发布目录全量发布（示例）：
```bash
python3 /home/source/Jetbrains/Probject/Gjj/scripts/remote_deploy.py <region>
python3 /home/source/Jetbrains/Probject/Gjj/scripts/remote_exec.py <region> "kubectl rollout restart deployment <dep1> <dep2> ... -n <namespace>"
```
3. 上传去重规则：
- `remote_deploy.py` 会在上传前校验目标 `host + port + default_path` 与本次文件签名。
- 若与最近一次成功上传记录一致，则自动跳过重复上传，避免同目标路径重复传包。

## 5. 前端打包与发布流程

### 5.1. 代码更新
```bash
cd <地区根目录>/WebstormProjects/capinfo-gjj-frontend-jshs-gm
git pull
```

### 5.2. 环境配置与验证
```bash
export PATH="/home/source/.vfox/cache/nodejs/v-16.20.2/nodejs-16.20.2/bin:$PATH"

node -v  # 应输出 v16.20.2
npm -v   # 应输出 8.x
```

### 5.3. 编译打包
清理历史构建文件并重新安装依赖后打包：
```bash
rm -rf dist
npm install
npm run build
```

### 5.4. 归档制品
1. 将 `dist` 目录打包成 `capinfo-gjj-frontend-jshs-gm.zip`。
2. 如果是单独发布前端，先清理 `<地区根目录>/release` 目录下旧的前端压缩包。
3. 将该压缩包迁移至 `<地区根目录>/release`。
```bash
zip -r capinfo-gjj-frontend-jshs-gm.zip dist
find <地区根目录>/release -maxdepth 1 -type f -name "capinfo-gjj-frontend-jshs-gm.zip" -delete
cp capinfo-gjj-frontend-jshs-gm.zip <地区根目录>/release/
```

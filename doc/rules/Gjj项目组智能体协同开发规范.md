# Gjj 项目组智能体协同开发规范

## 1. 目的
- 在既有“需求 -> 分析 -> 设计 -> 开发 -> 测试 -> 发布 -> 复盘”流程基础上，建立可持续、可交接、可审计的智能体协同研发机制。
- 解决多会话开发中的上下文丢失、过早宣布完成、验证不足、进度不可追溯等问题。

## 2. 适用范围
- 适用于 Gjj 项目组内所有 AI 协同研发任务与跨人接力任务。
- 适用于后端服务、前后端联调、工具脚本、自动化任务等研发场景。

## 3. 核心原则
- 增量交付：每轮会话只完成一个可验证特性。
- 状态外化：进展、决策、验证结果必须落盘，不依赖会话记忆。
- 先验后标：未完成验证不得标记通过。
- 清洁收尾：每轮会话结束时仓库保持可继续开发状态。
- 可回滚：关键变更必须具备回滚路径与执行步骤。

## 4. 标准工件（强制）
- 需求规格：`app_spec.txt`
- 特性清单：`feature_list.json`
- 会话进度：`claude-progress.txt`（记录结构参照 `doc/rules/claude-progress-template.txt`）
- 工作日志：`daily_work_log.md`（记录每日工时、工作类型与内容，用于填报工时系统）
- 启动脚本：`init.sh`
- 发布清单：`doc/rules/release-checklist.md`
- 复盘模板：`doc/review/postmortem.md`

## 5. 流程阶段与门禁

### 5.1 需求阶段
- 输出冻结需求范围、验收标准、非目标范围。
- 门禁：验收标准未明确不得进入分析阶段。

### 5.2 分析与计划阶段
- 将需求拆解为可验证特性，写入 `feature_list.json`，初始状态统一为 `passes: false`。
- 对拆解出的每项任务/特性进行**工时估算**，并排定工作计划（预估开始和完成时间），一并记录在 `feature_list.json` 或 `claude-progress.txt` 中。
- 门禁：特性不可验证、描述模糊、无优先级、无工时估算与计划不得进入设计阶段。

### 5.3 设计阶段
- 产出实现方案、项目骨架与 `init.sh`。
- 门禁：无启动路径、无回滚路径不得进入开发阶段。

### 5.4 开发阶段（会话循环）
- 每轮开始前必须读取规格、特性清单、会话进度、近期提交记录。
- 每轮只能处理一个最高优先级且未通过的特性。
- 每轮结束必须更新会话进度并提交代码。

### 5.5 测试阶段
- 先回归已通过核心能力，再验证本轮新增能力。
- 必须覆盖正常流、异常流、边界流；有 UI 场景时必须覆盖端到端验证。
- 门禁：验证不通过不得将 `passes` 置为 `true`。

### 5.6 发布阶段
- 发布前必须完整执行 `doc/rules/release-checklist.md`。
- 门禁：清单未完成、回滚不明确、关键风险未闭环不得发布。

### 5.7 复盘阶段
- P0/P1、关键故障、回滚事件必须使用 `doc/review/postmortem.md` 落盘复盘。
- 门禁：未形成防复发措施与责任人，不得关闭事件。

## 6. 会话执行协议（强制）
- 会话启动动作：
  - 读取 `app_spec.txt`、`feature_list.json`、`claude-progress.txt`。
  - 查看最近提交记录并执行基础回归验证。
- 会话执行动作：
  - 实现单特性闭环，边改边验。
  - 若发现回归问题，先修复后继续新特性。
- 会话收尾动作：
  - 更新进度与交接信息。
  - 更新特性状态。
  - **产出每日工作日志**：每日下班/收尾时，根据当日实际执行情况产出简洁明了的工作日志（如 `daily_work_log.md`）。日志内容必须包含工时填报系统所需字段：**所属项目名称、工作类型、耗费工时（小时）、工作内容描述（500字以内）**，以便直接复制填报工时系统。
  - 提交变更并记录验证证据。

## 7. 完成定义（DoD）
- 目标特性完成端到端验证。
- `feature_list.json` 状态已正确更新。
- 质量门禁通过（lint/typecheck/test，按项目实际执行）。
- 会话进度与交接记录完整。
- 发布任务附带发布清单，故障任务附带复盘文档。

## 8. 禁止项
- 禁止未验证即标记特性通过。
- 禁止删除或改写既有特性描述与步骤（仅允许更新 `passes` 状态）。
- 禁止在会话结束时留下不可运行或不可交接的代码状态。
- 禁止缺失进度文档与验证证据直接宣告完成。

## 9. 评审与审计
- 发布评审必查：发布检查清单、验证证据、回滚方案。
- 过程评审必查：会话进度、特性状态变更记录、提交留痕。
- 事故评审必查：复盘报告、防复发措施、责任人与截止日期。

## 10. 生效说明
- 本规范发布后即生效，作为项目组研发流程的强制组成部分。
- 若与模块私有约定冲突，以本规范与项目组总规范为准，私有约定需显式补充说明。

## 11. 增量共享规则（与既有规范去重后）
- 以下规则用于补齐既有规范未明确约束的协作与提交流程，不重复 DDD、命名、接口、异常、测试门禁等已存在条款。
- 提交前必须先同步远端主线分支并完成冲突处理，再执行本地提交，避免把冲突留给后续环节。
- 提交说明中必须固定追加“（AI生成提交）”，用于审计检索与来源识别。
- 代码变更必须先保留差异证据（如 MR Diff、变更清单或补丁记录），再写入目标文件，确保过程可追溯。
- 低风险变更默认由智能体直接执行并留痕；高风险变更必须先给出风险说明、影响面与回退方案后再进入人工确认。
- 跨会话与跨角色协作场景统一使用简体中文输出关键结论、交接信息与验证结果，保证团队理解一致与知识沉淀可复用。

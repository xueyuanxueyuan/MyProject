# Gjj 项目组前后端项目开发提示词

## 1. 适用范围
- 适用于 `Gjj` 工作区内当前已识别的前后端项目，重点包括：
  - 后端聚合工程：`zaozhuang/IdeaProjects/capinfo-gjj-busi-jshs`
  - 前端工程：`zaozhuang/WebstormProjects/capinfo-gjj-frontend-jshs-gm`
- 适用于新增功能、缺陷修复、重构优化、代码评审、提测前自检、智能体协同研发等场景。

## 2. 项目结构蒸馏
### 2.1 后端项目结构
- 后端采用 `Maven` 多模块聚合结构，聚合根为 `capinfo-gjj-busi-jshs`。
- 当前核心业务模块包括：
  - `capinfo-gjj-busi-zjjs-ywgl`
  - `capinfo-gjj-busi-zjjs-zhgl`
  - `capinfo-gjj-busi-zjjs-lcgl`
  - `capinfo-gjj-busi-zjjs-sp`
  - `capinfo-gjj-busi-cwhs-jzgl`
  - `capinfo-gjj-busi-jshs-gm-agg`
  - `capinfo-gjj-busi-jhgl-zjjh`
  - `capinfo-gjj-busi-zjjs-zbkh`
  - `capinfo-gjj-busi-zjjs-core`
- 每个业务模块通常拆分为：
  - `*-basic-svc-api`：对外接口、Feign、DTO、常量
  - `*-basic-svc-api-v2`：V2 接口和 DTO
  - `*-basic-svc-app`：启动类、环境配置、应用装配
  - `*-basic-svc-busi`：Controller、Service、Domain、DAO、Mapper、业务实现

### 2.2 前端项目结构
- 前端采用 `Vue 2 + Vue CLI + Element UI + Vuex + Vue Router + Axios`。
- 当前目录分层比较明确：
  - `src/api`：接口封装
  - `src/components/common`：公共基础组件
  - `src/components/<业务域>`：业务页面与业务组件
  - `src/common`：通用方法、权限、字典、请求封装、指令
  - `src/router`：路由
  - `src/store`：全局状态
  - `src/assets`：静态资源与全局样式

## 3. 代码放置硬约束
### 3.1 总原则
- 新增代码前，必须先判断它属于哪个业务模块、哪一层、哪个工程，再开始编码。
- 严禁为了图省事把业务代码、工具代码、接口定义、配置、样式、SQL、DTO、共享逻辑随意乱放。
- 如无法明确代码应该放在哪个模块或目录，必须先做结构分析并给出放置方案，再编码。

### 3.2 后端代码放置规则
- 新增 `DTO`、Feign 接口、Feign 降级、常量、接口契约，放在对应业务模块的 `*-basic-svc-api` 或 `*-basic-svc-api-v2`。
- 新增启动类、环境配置、装配入口、应用层启动相关代码，放在对应业务模块的 `*-basic-svc-app`。
- 新增 Controller、业务 Service、领域服务、业务实现、订阅处理器、JobHandler、DAO、Mapper、领域实体，放在对应业务模块的 `*-basic-svc-busi`。
- 新增数据库持久化对象、Mapper 接口、XML SQL，必须与当前业务模块保持同域放置，不允许跨模块乱放。
- 只有确实属于跨模块复用的通用能力，才允许放入 `capinfo-gjj-busi-zjjs-core` 或公共基础模块；否则必须优先放在业务模块内。
- 业务需求归属 `ywgl`、`zhgl`、`lcgl`、`sp`、`cwhs`、`jhgl`、`zbkh` 中哪一类，就必须进入对应模块，不允许放到“看起来最方便”的模块。

### 3.3 前端代码放置规则
- 新增接口定义，必须放在 `src/api/<业务域>`，禁止把请求直接写进页面组件。
- 新增页面和业务组件，放在 `src/components/<业务域>`。
- 只有被多个业务域复用的组件，才允许放入 `src/components/common`；单业务页面专用组件不得随意放到 `common`。
- 新增通用方法、权限、字典、请求封装、指令时，才允许放入 `src/common`；页面私有逻辑不得塞到通用目录。
- 新增路由配置，统一放到 `src/router`。
- 新增全局共享状态，统一放到 `src/store`；页面局部状态不要滥用全局状态。
- 新增静态资源和样式变量，放到 `src/assets`；业务逻辑代码不得混入资源目录。

### 3.4 放置前判断要求
- 编码前必须先输出“本次新增/修改代码的目标模块、目标层级、目标目录”。
- 如果涉及公共层改动，必须说明为什么不能放在业务模块内。
- 如果涉及跨模块复用，必须证明它是“真实复用”，而不是“提前抽象”。

## 4. 规范蒸馏出的提示词硬约束
### 4.1 文档与流程约束
- 接到任务后，先判断任务属于需求、分析、设计、开发、评审、发布、复盘中的哪个阶段。
- 分析类任务必须输出分析文档，并落盘到 `doc/需求分析/`。
- 评审类任务必须输出评审文档，并落盘到 `doc/评审记录/`；评审结论必须包含认可项、问题项、处理建议。
- 中大型需求或重构任务，在实质编码前必须先产出 SDD 设计文档，并落盘到 `doc/设计文档/`。
- 项目组公共规则、约定、提示词只能放在 `doc/项目规范/`、`doc/提示词/`，不能乱写到项目私有目录。

### 4.2 后端实现约束
- 后端开发必须遵循 DDD 分层思想，至少明确接口层、应用层、领域层、基础设施层的职责边界。
- 禁止 `controller` 直接调用 `dao/mapper`，禁止领域对象直接依赖 Feign、DB、MQ、Redis 等基础设施实现。
- `DTO` 只作为接口层和应用层边界对象，`Entity` 只在领域层和应用层内部流转，`DO/PO` 只在持久化层使用。
- 若历史模块仍是传统 `service + dao + entity` 结构，新需求也必须按增量治理思路逐步向 `domain/repository` 收敛。
- 接口统一返回 `R<T>`，禁止返回裸对象。
- 核心接口必须补齐参数校验、必要日志和接口注解。
- 业务异常统一使用 `BizException` 或项目等效业务异常，严禁吞异常、空 catch、模糊报错。
- Feign 降级禁止 `return null`，必须返回明确失败结果或抛出明确异常。
- 简单 CRUD 优先使用现有 MyBatis-Plus 方式，复杂查询放 Mapper XML，禁止随意拼接 SQL。
- 定时任务只负责触发，具体逻辑必须下沉到业务服务层。

### 4.3 命名与编码约束
- 变量、方法、类、接口、包名统一遵循“中文语义拼音缩写 + 驼峰命名”。
- 类名、接口名使用大驼峰；变量、方法使用小驼峰；DTO/DO/VO/Entity/Enum 后缀必须保留。
- 禁止使用无语义命名，如 `tmp1`、`abc`、`testObj`、`doProcess`。
- 同一业务域内缩写口径必须统一，新增缩写应先定义再使用。
- 代码修改坚持最小可用改动，避免无业务价值的大规模重命名和结构迁移。

### 4.4 配置、工具链与安全约束
- 编译、打包、发布前必须先切换并校验项目要求的 SDK 版本。
- 后端 Java 项目必须先执行 `vfox use java@17.0.2+8`，并通过 `java -version`、`mvn -v` 校验版本。
- 前端项目必须使用规范要求的 `Node.js`、`yarn` 版本和仓库源配置。
- 外部地址、开关、阈值、密钥、账号、证书等必须走配置与安全通道，禁止硬编码入代码。
- 未经明确要求，不得随意修改配置文件并纳入交付。

### 4.5 质量门禁与交付约束
- 缺陷修复必须以“不引入其他问题”为前提完成，提交前必须编译通过并完成关键回归验证。
- 开发完成后必须输出影响范围、验证清单、风险点与回滚思路。
- AI 协同任务必须同步更新进度、验证结论和交接信息，不能只在会话中口头说明。
- 未完成验证、未完成自检、未留存验证证据时，禁止宣告“完成”或“可发布”。

## 9. 提交前自检提示词
请按 `Gjj` 当前项目结构，对本次改动做提交前自检，并逐项输出“通过/不通过/需补充说明”：
- 是否已明确本次改动属于哪个项目、哪个模块、哪个层级；
- 是否所有新增代码都放在正确目录，没有乱放；
- 后端 `DTO/Feign/常量` 是否放在 `api/api-v2`；
- 后端业务实现是否放在 `busi`，配置与启动是否放在 `app`；
- 前端接口是否放在 `src/api/<业务域>`；
- 前端页面和业务组件是否放在 `src/components/<业务域>`；
- 是否避免把单业务逻辑错误放入公共层；
- 是否未改变原业务逻辑且未影响其他业务；
- 是否已按任务类型补齐 `doc/需求分析/`、`doc/设计文档/`、`doc/评审记录/` 等落盘文档；
- 后端是否遵循 `R<T>`、DTO、BizException、Feign 降级、DDD 分层等规范；
- 编译前是否已按要求切换并校验 SDK 版本；
- 是否已完成关键正常流、异常流、边界流验证；
- 是否确认业务逻辑完整可用后再交付。

# capinfo-gjj-frontend-jshs-gm 前端开发提示词

## 1. 适用说明
- 适用于 `capinfo-gjj-frontend-jshs-gm` 前端项目。
- 适用于新增功能、缺陷修复、重构优化、代码评审、提测前自检等场景。
- 项目技术栈：`Vue 2 + Vue CLI + Element UI + Vuex + Vue Router + Axios`。

## 2. 通用主提示词
请作为 `capinfo-gjj-frontend-jshs-gm` 项目的资深前端开发工程师开展工作，严格遵循当前仓库的真实实现方式，而不是套用通用前端模板。

项目背景与约束如下：
1. 当前项目是 `Vue 2 + Vue CLI + Element UI + Vuex + Vue Router + Axios` 架构。
2. 目录分层必须遵循现有结构：
   - `src/api`：接口封装层
   - `src/components/common`：公共基础组件层
   - `src/components/<业务域>`：业务页面与业务组件层
   - `src/common`：通用能力层
   - `src/router`：路由层
   - `src/store`：状态管理层
   - `src/assets`：静态资源与样式层
3. 页面开发必须优先复用现有公共能力，尤其是 `x-table`、`x-dialog`、`cap-form`、各类选择器和统一请求封装，避免重复造轮子。
4. 所有 HTTP 请求必须统一通过 `@/common/http` 发起，禁止在页面组件中直接创建 `axios` 实例。
5. 表格型页面优先沿用当前项目的配置驱动模式，将查询条件、列定义、工具栏和行操作集中在 `config` 中维护。
6. 命名保持当前项目风格，变量和方法使用驼峰命名法，组件必须显式声明 `name`。
7. 新增代码必须与当前项目编码风格保持一致，优先最小改动，避免无业务价值的大规模重构。
8. 如果涉及重构，必须保证原业务逻辑、接口契约、数据口径和页面交互结果不变，且不能影响其他业务。
9. 如果修改公共组件、公共方法、共享配置或共享接口，必须先评估影响范围，并说明需要回归的业务页面或功能点。
10. 复杂逻辑、兼容逻辑和公共能力改造需要补充清晰注释，重点解释原因、边界和风险。
11. 修改完成后必须执行自检，至少确认：
   - 需求是否完整实现；
   - 原有业务逻辑是否保持不变；
   - 关键正常流、异常流、边界流是否完整；
   - 共享模块改动是否影响其他业务；
   - 页面是否仍然可用，关键交互是否正常。
12. 输出结果时请始终给出：
   - 修改文件清单；
   - 核心实现思路；
   - 风险点与影响范围；
   - 验证步骤与验证结果；
   - 如未完成自动化验证，请明确说明人工验证范围。

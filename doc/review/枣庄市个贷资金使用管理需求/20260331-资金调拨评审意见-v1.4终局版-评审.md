# 枣庄市个贷资金使用管理需求-资金调拨评审意见-v1.4终局版

> **文档适用范围**：Gjj 工作区 `zaozhuang` 项目个贷资金调拨模块
> **评审日期**：2026-03-31
> **文档定位**：基于 v1.3 的待确认差异项，给出具备开发底线防御性质的最终裁决建议，可作为详细设计与测试用例的直接输入。

## 1. 认可项

1. **审计视角与证据链（保留自 v1.2/v1.3）**：高度认可将需求拆分为“直接证据”、“推导”和“待确认”三类，极大降低了研发对模糊需求的背锅风险。
2. **会议闭环工具（保留自 v1.3）**：认可引入“决策闭环表”和“术语统一表”，有效解决了评审会后无人拍板、前后端命名分裂的问题。

## 2. 问题项（针对 v1.3 待确认争议的漏洞剖析）

1. **问题 1：取消操作缺乏资源释放强约束**。v1.3 建议“申请”态允许取消，但未规定系统行为。如果单据取消但没有释放占用的“资金缺口清单”和“调拨额度”，将导致这些资金死锁，业务无法重新发起调拨。
2. **问题 2：金额字段定性过于软弱**。v1.3 将金额精度与负数校验交由产品决定。在金融级调拨中，允许金额为 0 或使用浮点数存储是极高危的资损隐患，不能留给非技术角色去“自由发挥”。
3. **问题 3：部分失败状态回写未给出兜底方案**。v1.3 提出需要状态机分支，但未直接给出标准解法。若产品要求“部分失败时整单重试”，极易引发已成功明细的二次重复打款。
4. **问题 4：资金安全降级风险**。v1.3 将“幂等与防重”放在 D 清单（工程增强建议）并依赖升级机制。但对于实际发生真实资金转移的“调拨支付”动作，幂等属于生命线，决不能作为可裁剪的增强项。

## 5. 处理结论/建议（终局裁决底线）

1. **资源强一致性释放**：强制规定任何产生“终态”（如作废、取消）的操作，必须在同一个本地事务内释放关联的业务单据锁定状态，允许其进入下一次调拨池。
2. **金额底线校验架构固化**：无需产品确认，直接由后端强制规定：**调拨金额必须 > 0，精度强制为两位小数，存储与计算强制使用 `BigDecimal`**。
3. **明细级重试法则**：明确“部分成功/部分失败”的总单据状态定义，且**仅允许对状态为“失败”的子明细发起重试或废弃**，从前端交互到后端接口，严禁提供“整单重试”入口。
4. **幂等校验升级为硬需求**：将“调拨支付的逐条幂等标识与防重放”从 D 清单强制划拨至 A 清单（硬需求），作为提测的阻断级前置条件。

## 6. 差异项/分歧项/变更项（v1.3 待确认项终局裁决表）

| 差异点 | v1.3 当前口径（建议） | v1.4 评审口径（系统底线约束） | 处理结论 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **`申请`状态是否允许取消** | 建议允许（逻辑推导） | 允许取消。但必须附带强约束：**取消即作废，必须同步释放占用的调拨额度与底层缺口单据** | 作为硬约束落入详细设计 | 强制执行 |
| **调拨金额类型与精度** | 由产品确定精度，架构固化类型 | **拒绝产品自定义**。后端强制执行：金额 > 0，禁止负数/零，精度 2 位小数，底层采用 `BigDecimal` | 作为公共校验拦截 | 强制执行 |
| **附件查看落点** | 在详细设计补充页面入口 | 在详细设计补充入口，但**强制要求复用系统现有全局附件组件**，禁止另起炉灶开发新控件 | 限制前端开发边界 | 强制执行 |
| **部分失败状态回写与重试** | 补状态机分支与用例编号 | 若发生部分失败，总单状态为“部分成功”，**仅允许对失败明细单独发起重试**，严禁整单重发 | 作为核心状态机逻辑 | 强制执行 |
| **幂等/防重的验收级别** | 归属 D 清单，靠触发条款升级为阻断 | 涉及资金转移，**强行从 D 清单踢出，并入 A 清单（硬需求）**，作为提测阻断项 | 变更需求验收基线 | 强制执行 |

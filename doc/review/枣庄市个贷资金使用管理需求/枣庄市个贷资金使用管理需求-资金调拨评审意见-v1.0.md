# 枣庄市个贷资金使用管理需求-资金调拨分析文档

## 1. 分析范围

本文基于《枣庄市个贷资金使用管理需求》中的以下范围进行分析：

- 1.1.1.2 贷款发放资金确认中的“资金调拨申请”相关要求
- 1.1.1.3 贷款资金调拨管理（申请后全流程处理）

## 2. 分析结论摘要

- 主业务链路明确：余额不足时，先按提交放款时间顺序处理可满足部分，再对不足部分发起调拨申请。
- 申请模式明确：余额不足账户采用集中调拨申请，而非逐笔贷款单独申请。
- 全流程明确：覆盖申请、审批、调拨支付、结算回写与终态。
- 验收重点明确：查询条件、按钮可点击条件、字段默认值与可编辑性、状态流转条件必须落成可执行清单。

## 3. A 清单：硬需求清单（逐条可验收）

以下仅保留需求原文可追溯约束，不含实现推导。

### 3.1 资金确认阶段（1.1.1.2）

1. 查询输入约束
- 贷款银行：可选，下拉列表
- 提交日期范围：必填，默认当前日期

2. 列表输出字段
- 贷款银行
- 放款银行账号
- 待发放汇总金额
- 当前账户余额
- 账户备付金
- 可用放款金额
- 操作（详情、发放确认）

3. 计算与显示口径
- 可用放款金额 = 当前账户余额 - 账户备付金
- 若可用放款金额 < 0，则显示 0

4. 操作可点击条件
- 放款确认按钮仅在可用放款金额 <> 0 时可点击

5. 放款确认规则
- 若当前账户余额 - 账户备付金 >= 待发放汇总金额：可直接确认全部放款
- 若当前账户余额 - 账户备付金 < 待发放汇总金额：
  - 按贷款申请提交时间顺序，优先生成已满足金额部分的放款确认
  - 未放款部分可发起资金调拨申请
  - 未放款放款申请清册作为资金调拨附件
  - 当前全部余额不足账户集中进行资金调拨申请

6. 资金调拨申请输入输出（在 1.1.1.2 中）
- 调拨清册字段：
  - 调拨收款银行：反显，不可修改
  - 调拨收款账号：反显，不可修改
  - 调拨付款银行：根据付款账号反显，不可修改
  - 调拨付款账号：下拉列表，可修改，默认资金余额表中可用余额最大银行
  - 调拨金额：文本框，可修改，默认缺口资金
  - 操作：增加行、删除行
- 资金余额统计表字段：
  - 银行编号、银行名称、银行账号、当前账户余额、账户备付金、待放款金额、可用金额

### 3.2 贷款资金调拨管理阶段（1.1.1.3）

1. 查询输入约束
- 状态：可选，下拉列表
- 申请日期范围：必填，默认当前日期

2. 列表输出字段
- 申请编号、申请日期、申请人、执行人、调拨总金额、状态、操作

3. 列表操作
- 修改、提交审批、详情、调拨支付、取消申请

4. 详情信息
- 调拨清册
- 资金余额统计表
- 审批记录（次序、节点、执行人、审批记录、开始时间、结束时间）

5. 驳回后处理规则
- 可修改后重新提交审批
- 也可取消申请，取消后申请作废，不可再修改或提交审批

6. 调拨支付输出字段
- 收款银行、收款账号、付款银行、付款账号、调拨金额、状态

7. 状态定义
- 申请、审批中、驳回、作废、待调拨、调拨确认、结算中、调拨完成

## 4. B 清单：状态流转表

| 状态 | 进入条件 | 可执行操作 | 退出条件 |
| --- | --- | --- | --- |
| 申请 | 驳回后完成修改且未重新提交 | 提交审批、取消申请、详情 | 提交审批后进入审批中；取消后进入作废 |
| 审批中 | 提交审批后至终极审批完成前 | 详情 | 审批通过进入待调拨；审批驳回进入驳回 |
| 驳回 | 审批驳回后未修改 | 修改、详情 | 修改后进入申请 |
| 作废 | 驳回后取消申请 | 详情 | 终态 |
| 待调拨 | 完成终极审批且未发起调拨支付 | 调拨支付、详情 | 发起调拨支付后进入调拨确认 |
| 调拨确认 | 发起调拨支付后未完成确认 | 详情 | 调拨支付确认后进入结算中 |
| 结算中 | 调拨支付确认后至结算返回前 | 详情 | 全部交易成功后进入调拨完成 |
| 调拨完成 | 调拨结算全部交易成功 | 详情 | 终态 |

## 6. D 清单：实现增强建议（非硬需求）

以下为工程实现增强建议，不作为原始需求硬约束。

1. 幂等与防重
- 调拨支付逐条交易设置幂等标识，防止重复扣划
- 重试场景需识别“同一业务请求”并返回一致结果

2. 失败治理
- 记录交易失败原因码、失败信息、重试次数、最近重试时间
- 支持失败条目修改后重新发起，保留前后版本链路

3. 并发与一致性
- 支付发起前进行付款账号可用余额二次校验
- 避免审批通过到支付执行期间因余额变动导致超额调拨

4. 审计与追溯
- 全链路留痕：申请、修改、提交审批、取消、支付发起、支付确认、结算回写
- 附件与调拨明细双向可追溯，满足审计与问题回溯

## 7. 验收建议

- 以 A 清单为需求验收基线，保证“是否满足原始需求”可直接判定。
- 以 B+C 清单形成测试用例输入，覆盖状态与字段口径。
- 以 D 清单作为技术方案评审项，独立评估是否纳入当前迭代。

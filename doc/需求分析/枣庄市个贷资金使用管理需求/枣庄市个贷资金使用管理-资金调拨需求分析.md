# 枣庄市个贷资金使用管理——资金调拨需求分析

## 1. 分析范围

本文聚焦原需求文档中的以下两部分：

- 1.1.1.2 贷款发放资金确认（其中“资金调拨申请”相关能力）
- 1.1.1.3 贷款资金调拨（申请后续管理与支付）

## 2. 主业务链路

### 2.1 触发与分流逻辑

- 系统按银行汇总待发放贷款金额，并计算可用放款金额：`当前账户余额 - 账户备付金`
- 当可用放款金额大于等于待发放汇总金额时：可直接完成全部贷款放款确认
- 当可用放款金额小于待发放汇总金额时：
  - 先按贷款申请提交时间顺序，优先完成“已满足金额部分”的放款确认
  - 对余额不足未放款部分，发起资金调拨申请
  - 将未放款申请清册作为调拨申请附件

### 2.2 集中申请原则

- 资金调拨申请以“当前全部余额不足账户”进行集中发起
- 不按单笔贷款逐个独立发起调拨申请

## 3. 需求硬约束清单（可追溯原文）

### 3.1 查询与展示口径

- 贷款发放资金确认查询条件：
  - 贷款银行：可选
  - 提交日期范围：必填，默认当前日期
- 列表输出核心字段：
  - 贷款银行、放款银行账号、待发放汇总金额、当前账户余额、账户备付金、可用放款金额、操作
- 可用放款金额口径：
  - `当前账户余额 - 账户备付金`
  - 若结果小于 0，则展示 0
- 放款确认按钮可点击条件：
  - 可用放款金额不等于 0

### 3.2 资金调拨申请（在资金确认中发起）

- 调拨清册字段：
  - 调拨收款银行、调拨收款账号（反显，不可修改）
  - 调拨付款银行（根据付款账号反显，不可修改）
  - 调拨付款账号（下拉列表，可修改，默认资金余额表中可用余额最大银行）
  - 调拨金额（文本框，可修改，默认缺口资金）
  - 操作（增加行、删除行）
- 资金余额统计表字段：
  - 银行编号、银行名称、银行账号、当前账户余额、账户备付金、待放款金额、可用金额

### 3.3 贷款资金调拨管理（申请后续）

- 查询条件：
  - 状态（可选）
  - 申请日期范围（必填，默认当前日期）
- 列表字段：
  - 申请编号、申请日期、申请人、执行人、调拨总金额、状态、操作
- 详情包含：
  - 调拨清册
  - 资金余额统计表
  - 审批记录（次序、节点、执行人、审批记录、开始时间、结束时间）
- 调拨支付字段：
  - 收款银行、收款账号、付款银行、付款账号、调拨金额、状态

### 3.4 状态与规则

- 驳回后：可修改并重新提交审批，也可取消申请
- 取消后：申请作废，不可再修改、不可再提交审批
- 审批通过后：可发起调拨支付
- 调拨支付按明细逐条发起交易，失败条目可修改后重新发起
- 状态语义：
  - 申请、审批中、驳回、作废、待调拨、调拨确认、结算中、调拨完成

## 4. 状态流转分析（业务视角）

### 4.1 状态流转表

| 当前状态 | 进入条件 | 可执行操作 | 退出条件/下一状态 |
| --- | --- | --- | --- |
| 申请 | 驳回后已修改，未重新提交 | 修改、提交审批、取消申请、详情 | 提交后进入审批中；取消后进入作废 |
| 审批中 | 已提交审批，终审未完成 | 详情 | 审批通过进入待调拨；审批驳回进入驳回 |
| 驳回 | 审批退回且未修改 | 修改、取消申请、详情 | 修改后进入申请；取消后进入作废 |
| 作废 | 驳回后取消申请 | 详情 | 终态 |
| 待调拨 | 审批通过未发起支付 | 调拨支付、详情 | 发起支付后进入调拨确认 |
| 调拨确认 | 已发起支付，未完成确认 | 详情 | 支付确认后进入结算中 |
| 结算中 | 调拨支付确认后等待结算回执 | 详情 | 全部结算成功进入调拨完成 |
| 调拨完成 | 全部交易成功结算 | 详情 | 终态 |

## 5. 字段与校验关注点（需求内可落地）

- 金额字段统一数值格式展示，保持前后列表一致性
- 调拨金额支持手工调整，需保证为空/非数字时可识别并拦截
- 调拨付款账号变更后，调拨付款银行需联动反显
- 调拨清册允许增加、删除行，页面需实时体现调拨总金额变化
- 可用放款金额为负时强制按 0 展示，避免出现负可放款金额误导

## 6. 附件与追溯要求

- 未放款贷款申请清册需作为资金调拨申请附件进入审批流
- 在调拨详情或审批关联视图应可追溯查看附件，支撑审计与验收

## 7. 实现增强建议（非原文硬约束）

以下内容用于提升可用性与稳定性，不作为原始需求硬性条款：

- 支付交易增加幂等控制（避免重复提交造成重复调拨）
- 保留交易流水、失败原因、重试记录，支撑问题定位与审计
- 对并发修改与余额变化增加一致性保护，避免清册与余额快照错位
- 针对支付失败场景增加标准化用例：单条失败、部分成功、修改后重试、重复提交防重

## 8. 验收建议清单

- 清单 A：硬约束验收（字段、默认值、可编辑性、按钮可用条件）
- 清单 B：状态流转验收（每个状态可操作项与禁止项）
- 清单 C：金额口径验收（可用放款金额、缺口金额、调拨总金额）
- 清单 D：异常与重试验收（失败修改、重新发起、状态回写）

## 9. 评审采纳补充（基于 v1.0 评审意见）

### 9.1 已采纳补充项

- 将“硬需求”与“实现增强建议”严格分层，避免评审时范围争议
- 补充查询输入必填与默认值作为独立验收口径：
  - 提交日期范围/申请日期范围：必填，默认当前日期
  - 状态：非必填
- 补充放款确认可点击条件与显示口径作为独立验收项：
  - 可用放款金额 = 当前账户余额 - 账户备付金
  - 小于 0 显示 0
  - 可用放款金额不等于 0 时可点击放款确认
- 补充调拨支付结果固定展示字段口径：
  - 收款银行、收款账号、付款银行、付款账号、调拨金额、状态
- 补充附件追溯落点：
  - 未放款清册附件应在调拨详情或审批关联视图中可查看

### 9.3 评审分歧保留项

- 幂等、强校验、失败原因字段等技术增强项：保留为“实现增强建议”，不纳入硬需求判定
- 放款确认异常提示文案：需求原文未定义，建议在详细设计阶段补齐
- 调拨支付扩展字段（如失败原因列）：原文字段为必落地项，扩展字段按迭代范围评估

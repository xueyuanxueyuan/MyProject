# 开发规范合并分析

## 1. 分析目标
- 合并 `doc/项目规范/Gjj项目组开发规范.md` 与 `doc/项目规范/项目开发规范.md`。
- 避免项目组总规范与项目专项规范长期双份维护造成内容漂移。
- 在智能体协同开发规范中补充重构与自检相关强约束。

## 2. 两份规范的定位差异
| 文档 | 主要定位 | 当前问题 |
| --- | --- | --- |
| `Gjj项目组开发规范.md` | 项目组通用规则，偏后端、流程、DDD、接口与质量门禁 | 缺少前端专项章节，导致前端规范独立分散 |
| `项目开发规范.md` | 当前前端项目专项规范，偏 `Vue 2` 工程落地 | 与总规范分离维护，长期容易重复和失真 |

## 3. 合并策略
| 项目 | 处理方案 | 原因 |
| --- | --- | --- |
| 主文档选择 | 以 `Gjj项目组开发规范.md` 作为主文档 | 该文档是项目组总规范，天然适合承载统一规则 |
| 前端规范内容 | 作为 `14. 前端项目专项规范` 并入总规范 | 保留专项规范的完整性，同时纳入统一治理 |
| `项目开发规范.md` | 调整为专项快速入口与摘要文档 | 避免重复正文，降低后续双维护风险 |
| 协同开发规则 | 修改 `Gjj项目组智能体协同开发规范.md` | 让“重构不改业务、不能影响其他业务、代码完成后必须自检”成为执行门禁 |

## 4. 本次新增约束点
### 4.1 重构约束
- 重构项目代码时，必须保证原业务逻辑、接口契约、数据口径、页面交互结果不变。
- 如确需变更原行为，必须先补设计说明、影响范围和评审结论。

### 4.2 影响面约束
- 修改代码前必须评估影响范围。
- 涉及公共组件、公共方法、共享配置、共享接口时，必须显式列出受影响业务并执行针对性回归。

### 4.3 代码完成后的自检约束
- 代码写完后必须执行业务自检。
- 自检至少覆盖：需求实现完整性、原业务逻辑保持情况、关键分支与异常分支、上下游调用完整性、共享模块对其他业务的影响。
- 必须结合项目实际执行编译、静态检查、自动化测试或关键业务路径人工回归，确认业务逻辑完整可用后方可交付。

## 5. 预期效果
- 规范入口统一，降低团队查找成本。
- 前端和后端规范归口一致，便于审计和评审。
- 智能体协同开发在“重构安全性”和“交付自检”方面具备更明确的门禁约束。

# 规范蒸馏补充到提示词分析

## 1. 目标
- 将 `Gjj` 项目组开发规范与智能体协同开发规范中“可直接执行”的约束，进一步蒸馏为提示词条款。
- 让提示词不仅描述项目结构，还能直接约束智能体的开发、文档、验证和交付行为。

## 2. 本次补充蒸馏的规范内容
### 2.1 文档流程类
- 分析任务必须落盘 `doc/需求分析/`
- 评审任务必须落盘 `doc/评审记录/`
- 中大型需求或重构必须先产出 `doc/设计文档/` 下的 SDD
- 公共规则与提示词必须落在 `doc/项目规范/`、`doc/提示词/`

### 2.2 后端研发类
- DDD 分层边界
- `DTO` / `Entity` / `DO/PO` 的对象边界
- `R<T>` 统一返回
- `BizException` 统一业务异常
- Feign 降级禁止 `return null`
- MyBatis-Plus / Mapper XML 使用边界
- JobHandler 仅触发，逻辑下沉 Service

### 2.3 编码与命名类
- 中文语义拼音缩写 + 驼峰命名
- DTO/DO/VO/Entity/Enum 后缀保留
- 禁止无语义命名
- 最小可用改动原则

### 2.4 工具链与交付类
- 编译前使用 `vfox` 切换并校验 SDK
- 外部地址、开关、密钥禁止硬编码
- 缺陷修复必须以“不引入其他问题”为前提
- 交付前必须给出影响范围、验证清单、风险点和回滚思路

## 3. 蒸馏方式
- 将“规范描述语句”改写为“执行型提示词语句”。
- 将“原则性要求”改写为“开始编码前必须做什么、完成后必须输出什么”。
- 将“文档归档要求”直接改写为提示词输出项，避免智能体漏产出文档。

## 4. 结果
- 已更新 `doc/提示词/Gjj项目组前后端项目开发提示词.md`
- 提示词已覆盖项目结构、代码放置、规范约束、交付门禁四大类内容

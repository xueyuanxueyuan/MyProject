# 资金调拨模块详细设计-定版

> 版本日期：2026-04-01  
> 设计输入：`doc/analysis/20260401-资金调拨需求分析-详细定版.md`

## 1. 设计目标

| 项目 | 说明 |
| --- | --- |
| 目标 | 将需求分析中的 A-001~A-013 硬约束落为可编码、可测试、可审计的实现方案 |
| 范围 | 资金调拨申请、审批后支付、结算回写、失败释放与新单重发 |
| 非目标 | 银行清算系统改造、跨城市资金统筹策略、报表中心改造 |

## 2. 分层设计

| 分层 | 组件 | 职责 |
| --- | --- | --- |
| 接口层 | `ZjdbController` | 入参校验、权限、幂等请求头透传、响应封装 |
| 应用层 | `ZjdbAppService` | 流程编排、事务边界、调用领域服务与外部适配器 |
| 领域层 | `ZjdbJhEntity` `ZjdbDomainService` | 状态流转、金额规则、资源释放规则、结果状态计算 |
| 基础设施层 | `ZjdbRepositoryImpl` `ZhyeGateway` `ZhifuGateway` | 持久化、余额查询、支付发起、结算回执接入 |

## 3. 核心对象模型

| 对象 | 关键字段 | 说明 |
| --- | --- | --- |
| 调拨申请主单 `ZjdbSqDO extends BaseDO` | `sqbh` `bjzt` `jgzt` `zjqkze` | 双状态模型，承载单据生命周期 |
| 调拨明细 `ZjdbMxDO extends BaseDO` | `mxbh` `fkzh` `skzh` `dbje` `mxzt` | 明细级支付执行与结果追踪 |
| 领域实体 `ZjdbJhEntity extends BaseEntity` | `sqbh` `bjzt` `jgzt` | 承载状态流转核心规则 |
| 失败释放记录 `ZjdbSfRzDO extends BaseDO` | `sqbh` `mxbh` `sfje` `sfsj` | 终态释放审计留痕 |
| 支付幂等记录 `ZjdbMdRzDO extends BaseDO` | `ydz` `mxbh` `qqzy` `cljg` | 防重放与同键自动重投 |

### 3.1 基类继承约束

| 对象类型 | 必须继承 | 约束说明 |
| --- | --- | --- |
| 持久化对象（DO/PO） | `BaseDO` | 禁止重复声明 `id/creator/created_time...` 等11个公共字段 |
| 领域实体 | `BaseEntity` | 聚焦业务字段与规则，不重复公共字段 |
| DTO对象 | 基础DTO父类（如项目已有 `BaseDTO`） | 仅承载入参与出参，不承载持久化字段 |

## 4. 状态机设计

| 维度 | 状态 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| 办结状态 | 未办结 | 申请创建后 | 全部明细获得最终结果 |
| 办结状态 | 已办结 | 明细全部完成（成功/失败） | 终态 |
| 结果状态 | 全成功 | 所有明细成功 | 终态 |
| 结果状态 | 部分成功 | 部分成功部分失败 | 终态（失败缺口释放） |
| 结果状态 | 全失败 | 所有明细失败 | 终态（失败缺口释放） |

## 5. 关键流程设计

### 5.1 调拨申请提交流程

| 步骤 | 处理 | 规则 |
| --- | --- | --- |
| S1 | 读取未放款缺口清册 | 必须存在缺口才允许发起 |
| S2 | 生成调拨清册默认值 | 默认付款账号=可用余额最大银行 |
| S3 | 执行三层金额校验 | 行级/同付款账号汇总/整单汇总 |
| S4 | 保存申请主单与明细 | 初始状态：未办结+待调拨前置状态 |

### 5.2 调拨支付流程

| 步骤 | 处理 | 规则 |
| --- | --- | --- |
| P1 | 支付前实时查询付款账户可用余额 | 余额不足直接阻断 |
| P2 | 按明细生成幂等键并写入幂等日志 | 同键请求必须返回一致结果 |
| P3 | 发起支付请求 | 支持同键自动重投（限定次数与窗口） |
| P4 | 接收支付结果并回写明细状态 | 成功/失败均记录原因码 |
| P5 | 计算主单结果状态并办结 | 失败明细触发缺口释放 |

### 5.3 终态释放与新单重发流程

| 步骤 | 处理 | 规则 |
| --- | --- | --- |
| R1 | 进入终态（部分成功/全失败） | 触发释放流程 |
| R2 | 事务内释放失败明细缺口占用 | 与状态落终态同事务提交 |
| R3 | 写释放日志 | 可审计可追踪 |
| R4 | 允许业务员基于释放缺口发起新单 | 禁止在原单据内人工重试 |

## 6. 接口设计（建议）

| 接口 | 方法 | 入参重点 | 出参重点 |
| --- | --- | --- | --- |
| `/zjdb/apply` | POST | `ZjdbApplyReqDTO`（调拨清册、附件ID、日期范围） | `R<ZjdbApplyRespDTO>`（申请单号、初始状态） |
| `/zjdb/approveResubmit` | POST | `ZjdbApproveReqDTO`（申请单号、修改后清册） | `R<ZjdbApproveRespDTO>`（提交结果） |
| `/zjdb/pay` | POST | `ZjdbPayReqDTO`（申请单号） | `R<ZjdbPayRespDTO>`（明细执行摘要、主单双状态） |
| `/zjdb/detail` | GET | `ZjdbDetailReqDTO`（申请单号） | `R<ZjdbDetailRespDTO>`（清册、审批记录、附件、双状态） |
| `/zjdb/relaunch` | POST | `ZjdbRelaunchReqDTO`（原单号、失败明细映射） | `R<ZjdbRelaunchRespDTO>`（新单号） |

### 6.1 接口统一规范

| 项目 | 约束 |
| --- | --- |
| 返回格式 | 必须统一使用 `R<T>` 包装 |
| 入参对象 | 必须使用 `XxxReqDTO` |
| 出参对象 | 必须使用 `XxxRespDTO` |
| 领域隔离 | Controller 仅调用应用层，不直接依赖 Repository/DAO |

## 7. 数据校验与错误码

| 规则ID | 校验规则 | 错误码 | 错误文案 |
| --- | --- | --- | --- |
| V-001 | 调拨金额必须大于0且2位小数 | ZJDB_4001 | 调拨金额格式不合法 |
| V-002 | 行金额不得超过对应缺口 | ZJDB_4002 | 调拨金额超过缺口金额 |
| V-003 | 同付款账号汇总不得超过实时可用 | ZJDB_4003 | 付款账号可用余额不足 |
| V-004 | 整单汇总不得超过缺口总额 | ZJDB_4004 | 调拨总额超过缺口总额 |
| V-005 | 重复幂等键重放 | ZJDB_4005 | 请求重复，请勿重复提交 |

## 8. 事务与一致性

| 场景 | 事务策略 | 一致性要求 |
| --- | --- | --- |
| 取消/作废 | 本地事务 | 状态变更与资源释放原子提交 |
| 支付回写 | 本地事务+幂等日志 | 明细状态、主单状态、日志一致 |
| 终态释放 | 本地事务 | 失败缺口释放与办结状态一致 |

## 11. 回滚策略

| 场景 | 回滚动作 |
| --- | --- |
| 新规则上线异常 | 关闭调拨支付开关，仅保留查询与详情 |
| 幂等逻辑异常 | 启用只读保护，暂停支付入口并保留审计数据 |
| 状态回写异常 | 启动补偿脚本按明细重算主单双状态 |

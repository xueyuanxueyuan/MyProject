# 枣庄项目联调环境发布与生产环境上传记录

## 1. 任务背景
完成“手续费预提调整核心链路”的重构与修复后，需将最新的代码打包发布至枣庄的测试联调环境，并同步将打包产物上传至生产环境备用。

## 2. 打包过程 (Build)
根据兵部的规范与工具链约束，在枣庄项目根目录 (`zaozhuang/`) 执行了前后端独立打包与产物汇总：

### 2.1 后端打包
- **工具链**：通过 `vfox` 切换并注入了 `java@17.0.2+8` 和 `maven@3.9.14`。
- **构建命令**：`mvn clean package -DskipTests -s settings-tsp.xml`
- **执行结果**：所有微服务模块（包括修改的 `capinfo-gjj-busi-cwhs-jzgl-basic-svc-busi` 及之前出现编译错误的模块）均 `BUILD SUCCESS`，成功生成了最新的 `*-app.jar`。

### 2.2 前端打包
- **工具链**：通过 `vfox` 切换至 `nodejs@16.20.2`。
- **构建命令**：`npm install` -> `npm run build`
- **打包产物**：将生成的 `dist` 目录压缩为 `capinfo-gjj-frontend-jshs-gm.zip`。

### 2.3 产物聚合
- 清空了 `/zaozhuang/release` 目录。
- 将后端所有 `*-app.jar` 与前端的 `capinfo-gjj-frontend-jshs-gm.zip` 集中移动到了 `release` 目录中，共计 8 个构建产物。

## 3. 发布与上传 (Deploy)
使用项目规范中的发布脚本，基于 `config/deploy_servers.json` 的配置执行了远端操作。

### 3.1 发布到枣庄联调环境 (`zaozhuang:test-zjb`)
- **上传产物**：执行 `python3 scripts/remote_deploy.py zaozhuang:test-zjb`，将 `release` 目录下的 8 个产物全部上传至 `100.70.0.189` 服务器的 `/data/app/zzgjj-test/capinfo-gjj-jshs` 目录，上传成功。
- **服务重启**：执行 `python3 scripts/remote_exec.py zaozhuang:test-zjb "kubectl rollout restart deployment ... -n zzgjj-test-zjb"`，成功对相关的 8 个 deployment 触发了滚动重启。联调环境发布完毕。

### 3.2 发布到枣庄测试环境 (`zaozhuang:test`)
- **上传产物**：执行 `python3 scripts/remote_deploy.py zaozhuang:test`，脚本检测到由于测试环境与联调环境共享同一台主机的同一个路径 (`100.70.0.189:22022/data/app/zzgjj-test/capinfo-gjj-jshs`)，因此触发了内置防重传机制，直接复用刚才的文件。
- **服务重启**：执行 `python3 scripts/remote_exec.py zaozhuang:test "kubectl rollout restart deployment ... -n zzgjj-test"`，成功对 `zzgjj-test` 命名空间下对应的 8 个 deployment 触发了滚动重启。测试环境发布完毕。

### 3.3 上传到枣庄生产环境 (`zaozhuang:prod`)
- **上传产物**：执行 `python3 scripts/remote_deploy.py zaozhuang:prod`，将 8 个产物上传至生产服务器对应的目录 `/data/app/zzgjj-prod/capinfo-gjj-jshs` 中，上传成功。
- **备注**：依据指令，仅对生产环境执行了物理包“上传”操作，未触发生产环境的 K8s 容器重启。

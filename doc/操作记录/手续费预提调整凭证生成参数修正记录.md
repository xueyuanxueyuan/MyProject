# 手续费预提调整凭证生成参数修正记录

## 1. 任务背景
在《手续费预提调整核心链路重构与优化方案_V2.1.md》实施完成后，根据反馈要求，“手续费预提调整”的财务凭证生成逻辑应当与原有的“手续费预提”凭证生成逻辑（`gjsxfYtPzSc` 与 `gdSxfjsPzSc`）保持高度一致，以确保相同的业务场景下入账凭证结构与枚举常量一致。

## 2. 参照排查与修正点

通过对 `CwhsServiceImpl.java` 中现有的归集/个贷手续费预提凭证生成代码的排查，发现并修正了 `generateSxfyttzPz` 方法中的多处硬编码和遗漏：

### 2.1 实体枚举常量的替换
- `xtlx`（系统类型）：从硬编码 `"ZJJS"` 修改为标准枚举 `XtlxEnum.XTLX_ZJJS.getValue()`。
- `zhlx`（账户类型）：从硬编码 `"01"` 修改为标准枚举 `ZhlxEnum.ZHLX_HQ.getValue()`。
- `ywzjlx`（业务资金类型）：从硬编码 `"01"` 修改为标准枚举 `YwzjlxEnum.YWZJLX_YT.getValue()`。
- `jszjlx`（结算资金类型）：从硬编码 `"01"` 修改为标准枚举 `JszjlxEnum.JSZJLX_GJJ.getCode()`。
- `jslx`（结算类型）：从硬编码 `"01"` 修改为标准枚举 `Jslx.NZ.getCode()`。

### 2.2 特殊核算分类字段的补充
- **`hsfl2`（核算支行）**：参照 `gdSxfjsPzSc` 的逻辑，如果是“个贷手续费预提调整”，必须设置 `mxDTO.setHsfl2(currentMx.getZhbh())` 以绑定核算支行信息；如果是“归集手续费预提调整”，则保持为空字符串。

### 2.3 当前操作人获取方式的修正
- 之前代码错误使用了 `currentUser.getName()` 获取操作人姓名，导致编译失败（`CurrentUser` 泛型内部缺少该方法）。
- 现已参照其他财务记账方法的逻辑，修正为 `currentUser.getUserDetail().getName()` 安全获取。

### 2.4 日期与流水号处理逻辑对齐
- `jzrqStr`（记账日期）：弃用 `DateUtil.formatDate(new Date())`，改用与预提生成一致的统一日切日期获取方法 `cwhsService.getJzrq()`。
- `ywlsh` / `jslsh` / `ywslbh`（业务/结算流水号）：在预提生成中是调用分布式流水号工具（`lshTools`）动态生成，但考虑到预提调整业务在保存时前端已经明确下发了关联影像材料的流水号（`req.getYwlsh()`），因此保持直接沿用 `req.getYwlsh()`，以维持业务单据与凭证的关联溯源。

## 3. 验收结果
- **编译检查**：通过 `mvn clean compile` 验证，代码全部编译通过，依赖引入与枚举调用无误。
- **静态检查**：代码结构已与其他标准凭证生成函数高度吻合，消除了因手误带来的数据格式风险。

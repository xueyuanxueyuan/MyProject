# 手续费预提调整凭证主表发生额与总金额计算修正

## 1. 发现问题
在验收过程中，用户（架构师）指出代码中凭证主表（`CwjzYwlsDTO`）的 `Zje` 和 `Ywfse` 采用了绝对值汇总后“除以2”（`sumAbsTzce.divide(new BigDecimal("2"))`）的计算方式。这是一种为了处理“双边凭证”（同时包含借方和贷方明细的流水集合）时为了避免重复计算总额而采用的特殊算法。
然而，在“手续费预提调整”及“手续费预提”场景中，传入记账引擎的明细列表 `mxDTOList` 仅包含**单边业务明细**（即手续费预提科目的调整数据），另一边的财务科目由财务记账引擎 `cwhsService.receiveCwjzxx` 根据 `Ywlx` 自动补齐。因此，在这个地方“除以2”完全画蛇添足，会导致实际入账金额缩水一半。

## 2. 参照标准与修正策略
通过比对原有的《归集手续费预提凭证生成》方法（`gjsxfYtPzSc`）：
```java
// 原预提业务的正确算法
BigDecimal zjehj = list.stream().map(CwjzYwlsMxDTO::getZje).reduce(BigDecimal::add).orElse(BigDecimal.ZERO);
cwjzYwlsDTO.setZje(zjehj);
cwjzYwlsDTO.setYwfse(zjehj);
```

**修正动作：**
1. **移除绝对值累加循环**：删除了原代码中 `sumAbsTzce = sumAbsTzce.add(tzce.abs())` 的循环计算逻辑。
2. **采用流式求和**：直接通过 `mxDTOList.stream().map(CwjzYwlsMxDTO::getZje).reduce(BigDecimal::add)` 将组装好的所有明细 `Zje` 进行代数求和，得到汇总的增加额 `zjehj`。
3. **直接赋值主表**：将求得的 `zjehj` 直接赋值给主表的 `Zje` 和 `Ywfse`，完全与预提凭证的生成逻辑保持一致。

## 3. 验证结果
- 编译通过（BUILD SUCCESS），消除了潜在的发生额减半 BUG，代码结构更加清爽且符合单边凭证接口规范。

# 科目辅助项配置（kmfzx.vue）全选功能初始化 Bug 修复记录

## 问题背景
在之前的操作中，我们为科目辅助项配置页面（`kmfzx.vue`）的 `el-tree` 上方添加了“全选/半选”功能。但在使用中发现了一个小问题：**当原来的数据已经全部勾选过时，首次进入页面，顶部的全选框并没有回显为勾选状态**。

## 原因分析
1. **`el-tabs` 懒加载导致 `el-tree` 未挂载**：
   `el-tabs` 中的 `el-tab-pane` 内容在首次激活前是不会挂载到 DOM 树中的，这就导致 `addTab` 循环插入标签页时，除了激活的标签页，其他非激活标签页的 `el-tree` 组件完全不存在（即 `this.$refs['tree-'+item.name]` 为 `undefined`）。
2. **`el-tree` 组件的异步渲染**：
   即便对于当前激活的 `el-tab-pane`，`el-tree` 解析 `default-checked-keys` 并将其同步到内部状态中需要一定时间，仅仅依赖一次简单的 `$nextTick` 调用 `getCheckedNodes` 可能仍然取不到正确的选中节点。
3. **回退逻辑的脏数据问题**：
   针对未挂载的 `el-tree` 组件，原本直接对比 `item.checkedKeys.length` 与树节点总数 `totalCount` 的逻辑不够严谨：后端下发的数据（`checkedKeys`）可能包含废弃节点的 ID，或是只包含末级节点的 ID，从而导致长度比对失效，无法准确判断是否处于“全选”状态。

## 解决步骤
1. **显式传入 `isInit` 标识强制基于数据计算**：
   修改了 `addTab` 方法，直接调用 `this.handleTreeCheck(..., true)`，要求初始化时直接根据 `item.checkedKeys` 进行计算。并结合 `setTimeout` 双重兜底，以防树节点异步渲染不同步。
2. **重构计算全选的逻辑（剔除脏数据影响）**：
   修改了 `handleTreeCheck` 方法：
   - 使用递归遍历 `item.data` 获取出**当前树中实际拥有的所有节点 ID**（`allNodeIds`）。
   - 提取后台下发的 `item.checkedKeys` 与 `allNodeIds` 取交集，得出**有效勾选的节点列表**（`validCheckedKeys`）。
   - 最后对比 `validCheckedKeys.length` 和 `totalCount`（`allNodeIds.length`）。如果两者一致且大于0，则判定为“全选”，精准点亮全选框。

## 验证与留痕
- 现在即便是多个辅助项配置 Tab 首次加载，只要后台数据证明所有可用节点已被选中，对应的“全选” Checkbox 将会完美呈现勾选状态，无论是当前可见的 Tab 还是隐藏在后的 Tab。
